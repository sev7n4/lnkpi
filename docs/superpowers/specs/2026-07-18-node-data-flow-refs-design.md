## 0. 决策摘要


| 项              | 结论                                                                    |
| -------------- | --------------------------------------------------------------------- |
| 第一优先           | **数据贯通**：连线/上传/资产 → 下游 Dock 可引用；用户仍手动点各节点「生成」                         |
| 自动级联生成         | **不做**（引用变化只更新 Dock 预填与芯片，不自动重跑下游）                                    |
| `prompt` 角色    | **文本类上游**；优先用 `content`；功能单一：为下游图/文/视/音的dock-studio的提示词 **prompt 注入** |
| 多上游 UI         | 统一 **RefChip**；**单行横滑**；类型靠图标 + `T1/I1/V1/A1` 区分；支持 `@T1` 等（实现可分期） |
| 文本合并           | **Chat 大模型归纳合并**（非朴素字符串拼接）；见 §5.3 |
| 引用源            | 三类同源：① 连线上游节点 ② 资产库 ③ 本节点上传/拖拽 |
| 移除引用           | **移除芯片 = 断边**（或移除本地附件绑定）；不只藏 UI |
| 可连性矩阵细化 / 环检测  | **本期不做**，文档写清，后续迭代 |
| `prompt` 多风格长文 | 一期 **整篇灌入**；「选用某一风格段落」后续再做 |
| 积分             | 按**下游节点每次生成**计费；图→文本与普通文本同价；文本合并成本计入该次生成（不加价） |
| 音频对图片芯片       | 一期 **只展示**，生成侧可不消费 |
| 上游 → `prompt`     | **B**：长文芯片**不进入**模式生成上下文；仅可预填空的短需求框（见 §3.2 / §10） |
| 持久化            | **边 + 本地附件 id**；不把上游全文拷贝进下游（解析时实时读上游） |

## 1. 目标与非目标



### 1.1 目标

用户在画布上通过连线、资产库、上传/拖拽，把上游产出的**文本 / 图片 / 视频 / 音频**变成下游节点可感知的**引用列表**，并在 Dock Studio 中以统一芯片 UI 展示；点击生成时，按规则把引用合并进该次生成请求。

成功标准（验收口径）：

1. 多文本 → 一图：Dock 出现多个「文本」芯片（带序号），生成时提示词可带上这些文本内容（或 `@n` 展开）。
2. 多图 → 一图 / 一视频：全部进入参考图列表（非只取第一张）。
3. `prompt` → 图/文/视/音：下游注入的是 `content`**（长文）优先**，无则回退短 `prompt`。
4. 本地输入框与芯片**并存**；清空输入框不会清掉芯片。
5. 删芯片后对应连线断开（或本地附件解绑），刷新后不再出现。
6. 图 → 文本：支持「看图出方案」（见 §3.2），例如衣服照片 → 电商主图/详情/细节/模特方案。



### 1.2 非目标（本期明确不做）


| 能力                                                          | 说明                                  |
| ----------------------------------------------------------- | ----------------------------------- |
| 视 → 图                                                       | 抽帧作参考图等，后续                          |
| 视 → 文本                                                      | 视频理解 / ASR 转写，后续                    |
| 音 → 文本                                                      | ASR，后续                              |
| 音 → 音                                                       | 拼接 / 音色参考，后续                        |
| 音 → 视频                                                      | 配乐轨 / 对口型等，后续                       |
| 按源类型严格限制可连目标                                                | 产品写清矩阵意向，工程本期不强制拦截                  |
| 环检测与复杂 DAG 治理                                               | 后续                                  |
| 上游完成自动触发下游生成                                                | 不做                                  |
| `prompt` 内「点选某一风格段」灌入                                       | 后续；一期整篇                             |
| shot / sceneComposer / videoComposition / worldModel 专用编排增强 | 不在本期主路径；若被连入，按 §3 能映射的字段弱支持或忽略并文档说明 |

## 2. 「排序」澄清（具现例子 + 推荐默认）

你已理解「多个上游 = 参考列表 + 序号」。下面补充**序号从哪来、为什么要能调序**。

### 2.1 场景

下游是一个**图片节点**，连了三个上游：

1. 先连了文本节点 A（文案：「夏季新款连衣裙」）
2. 再连了图片节点 B（白底商品图）
3. 最后连了 `prompt` 节点 C（已生成多风格长文）

Dock 参考区为**单行**（可横滑），例如：

```text
[T1 文本·A] [I1 🖼 B] [T2 提示词·C]   ← 一排，靠前缀区分类型
```

后端/请求体用 `mediaType` + `groupIndex`（或直接 `refKey: "T1"`）区分，**不依赖 UI 分行**。

### 2.2 「边创建序」是什么

不拖拽时，默认按**连上该下游的时间先后**在**同类型内**编号：

- 先连 A → `T1`
- 再连 B → `I1`
- 再连 C → `T2`

用户若在输入框写：`主图跟 @I1，文案用 @T2`，系统按 `refKey` 解析。

### 2.3 「拖拽芯片排序」是什么

用户把芯片从 `A, B, C` 拖成 `C, B, A` 后：

- 同类型内序号重排（例如文本变成 C=`T1`、A=`T2`；图 B 仍为 `I1`）  
- **影响**：多文本送进合并 LLM 的先后、主参考图 `I1`、`@Tn` 指向

**不调序时**：仅边创建序也能用。  
**要调序的原因**：同一套连线，用户想改「谁算第一参考」而不想拆线重连。

### 2.4 已定规则

| 规则 | 结论 |
|---|---|
| 默认序号 | **边创建先后**（持久化后按 edge 存储顺序 + `refOrder` 覆盖） |
| 用户调序 | **支持单行内拖拽芯片**；结果写入 `data.refOrder: string[]` |
| 编号方式 | **已定 `T1/I1/V1/A1` 分组序号**（同类型内按 `refOrder` 编号） |
| 布局 | **单行横滑**，不按类型多排（省 Dock 高度） |
| `@` 语法 | `@T1` / `@I2`；一期可先整组自动带入，`@` 解析为 P1.1 |
| 对后端 | **无困扰**：请求传结构化 `refs[]`（含 `refKey`/`mediaType`/`payload`），与 UI 是否分行无关 |

## 3. 引用能力矩阵



### 3.1 符号

- ✅ 本期做（资源引用 + 生成注入）  
- 🧠 本期做且含「理解/生成」  
- ⏳ 后续  
- — 无意义或不做



### 3.2 矩阵（源类型 → 下游节点类型）


| 上游 \ 下游                      | text                       | prompt                                 | image     | video                           | audio                       |
| ---------------------------- | -------------------------- | -------------------------------------- | --------- | ------------------------------- | --------------------------- |
| **text**（`content`/`prompt`） | ✅ 注入为文本参考                  | ✅ 芯片可展示；**仅当本节点短需求为空时预填摘要**（不进模式生成上下文，见 §10） | ✅ 注入提示词   | ✅ 注入提示词                         | ✅ 注入提示词/口播稿                 |
| **prompt**（优先 `content`）     | ✅ 整篇注入文本参考                 | —（一般不链 prompt→prompt；不强制禁）             | ✅ 整篇注入提示词 | ✅ 整篇注入提示词                       | ✅ 整篇注入提示词                   |
| **image**                    | 🧠 **看图理解后参与文本生成**（见 §3.3） | ⏳ 可选后续（prompt 分类仍以用户短需求为主）             | ✅ 多图参考列表  | ✅ 多图参考 / 图生视频                   | ✅ 作封面/语境图引用（生成侧若暂不消费则仅展示芯片） |
| **video**                    | ⏳                          | ⏳                                      | ⏳ 抽帧      | ✅ 视频参考列表（URL/附件；消费能力随 provider） | ⏳                           |
| **audio**                    | ⏳ ASR                      | ⏳                                      | —         | ⏳                               | ⏳                           |


说明：

- **上游 → `prompt`（已定 B）**：`prompt` 节点生成**只吃本节点短需求字段 `prompt` + 模式模板**。上游文本/长文芯片：
  - Dock **可以展示**（便于用户看见来源）；
  - **仅当**本节点 `prompt` 为空时，可用上游摘要**预填一次**短需求框（用户改过后不再覆盖）；
  - **不**把长文送进 Call-1/Call-2，**不**对上游长文做合并 LLM 再扩写（避免套娃）。
  - 决策依据见 §10。
- **音频下游**：文本芯片可进 TTS prompt；图片芯片一期只展示。
- **图→文本积分**：与普通文本同价（已拍板）。



### 3.3 图 → 文本（🧠 特例）

用户故事：上游一张衣服图 → 下游文本节点生成「电商主图 / 详情图 / 细节图 / 模特图」方案。

行为：

1. Dock 出现图片芯片（可多张）。
2. 用户在文本节点输入框写需求（可空，空则用默认指令模板）。
3. 点击生成：后端/Agent **带图调用视觉能力**（或等价：先描述再文案），产出方案 Markdown/纯文本写入 `content`。
4. 积分：与普通文本生成同价（已拍板）。

本期不做：视频理解、音频转写。

## 4. 统一引用源模型（Ref）



### 4.1 三类源 → 同一 RefChip


| sourceKind | 来源                          | 删除芯片时                          |
| ---------- | --------------------------- | ------------------------------ |
| `edge`     | 画布边：`upstreamNodeId → this` | **删除对应 edge**                  |
| `asset`    | 从素材库拖入/应用绑到本节点              | 解除 `data.localRefs` 中该项（不断其它边） |
| `upload`   | 本节点上传/拖拽文件                  | 删除本地附件绑定（及存储对象策略按现有上传逻辑）       |




### 4.2 逻辑字段（产品级，非最终 DB schema）

每条引用至少包含：

```text
refId          稳定 id（edgeId / asset绑定id / uploadId）
mediaType      text | image | video | audio
sourceKind     edge | asset | upload
label          展示名（节点标题 / 文件名）
preview        文本摘要 | 缩略图 URL | 时长等
payload        生成时用：text 正文 / image url / video url / audio url
orderKey       排序（拖拽后更新）
groupIndex     组内序号 T1/I1/…
```

解析原则：

- **不**把上游全文冗余拷贝进下游持久化；`payload` 在打开 Dock / 点击生成时**实时解析**。  
- 上游节点被删：芯片显示失效态，生成时跳过并提示；用户点移除则清理绑定。



### 4.3 `prompt` 作为文本类上游

- `mediaType = text`  
- `payload.text = content?.trim() || prompt?.trim()`  
- 芯片图标可用「提示词」变体，与普通 text 区分（便于辨认），但仍属文本组编号。

## 5. Dock Studio UI 重构（产品要求）



### 5.1 信息架构（单行）

每个支持生成的 Dock（text / prompt / image / video / audio）在输入区上方增加 **单行「参考」条**（横向滚动，不换多排）：

```text
┌─ 参考（单行可横滑、可拖拽调序）─────────────────────┐
│ [T1 …] [I1 🖼] [T2 …] [V1 ▶] [A1 ♪]     →→        │
└──────────────────────────────────────────────────┘
┌─ 本地输入 ─────────────────────────────────────────┐
│  prompt / 文案输入框 …                               │
└──────────────────────────────────────────────────┘
│ 模型 …                         [🎤] [生成]          │
```

类型辨识：**图标样式 + `T/I/V/A` 前缀**，无需分行。后端只认结构化字段，与排版无关。

### 5.2 芯片交互

| 操作 | 行为 |
|---|---|
| 点击芯片 | 预览全文/大图（轻量 popover） |
| 拖拽 | **单行内**调序；按类型重算 `T1..` / `I1..` |
| 芯片上「×」 | 移除引用：**断边**或解绑本地/资产 |
| 拖入文件到节点或 Dock | 新增 `upload` 芯片 |
| 从资产库拖入 | 新增 `asset` 芯片 |

### 5.3 与「本地输入并存」及文本合并

- 本地输入 = 用户本节点正在编辑的主 prompt/文案。  
- 芯片 = 外部引用。  
- **废弃旧行为**：本地非空则完全忽略上游。

#### 5.3.1 图片 / 视频 / 音频（非文本）

1. **图片**：全部进入 `referenceImages[]`；**主参考图 = I1**（图生视频等只需单主图时用）。  
2. **视频 / 音频**：以 URL 列表进入请求元数据；provider 不支持则忽略并打日志，UI 仍保留芯片。  
3. **音频 Dock 上的图片芯片**：一期 **只展示**，不强制参与生成。

#### 5.3.2 文本类：大模型归纳合并（已定）

**不做**前端/服务端的朴素字符串拼接作为最终 prompt。

流程：

1. 收集文本芯片 `T1..Tn`（含 `prompt` 节点长文）及本地输入（若有），每条带 `refKey` + 短标签 + 正文。  
2. 若「有效文本条数」≤ 1 且无「多源冲突」（例如仅本地、或仅单个 T1）：**跳过合并 LLM**，该条直接作为下游生成 prompt。  
3. 若 ≥ 2 条文本源（多芯片，或 1 芯片 + 本地非空）：先调用一次 **Chat 合并**（system：按下游节点类型归纳成一条清晰可用的提示词/文案；保留关键约束；标明曾参考哪些 `Tn`），得到 `mergedText`。  
4. 再将 `mergedText` + `referenceImages[]` 等交给下游真正的生成（文/图/视/音）。  

例外：**下游为 `prompt` 节点时不适用本合并进生成**（决策 B）：不把上游长文合并进 Call-1/Call-2；见 §10。
积分：合并调用成本 **计入该次下游生成**，**不加收**（与「图→文本同价」一致的产品策略）。超时：合并 + 生成合计仍受该类型超时约束（实现计划可对合并单设较短上限）。

对后端：**无困扰**——推荐 API 形态为显式两段或服务端内聚：

```text
客户端提交：localPrompt + refs[{ refKey, mediaType, text|url, label }]
服务端：   (可选) mergeTexts(refs, local) → mergedText
          → generate*(mergedText, images…)
```

客户端不必自己拼长字符串；UI 单行也不影响协议。



### 5.4 `@` 引用（产品约定）

- 用户可在本地输入中写 `@T1`、`@I2`。  
- 一期实现可拆：先「整组自动带入」（无 `@` 也带齐），再增强为「仅展开被 `@` 的项 + 未提及的仍可作静默多模态附件」。  
- 规格默认：**一期整组自动带入**；`@` 高亮与解析作为 **P1.1**。

## 6. 连线与数据流（行为）



### 6.1 连上之后

1. 下游 Dock 立即出现对应芯片（实时解析上游）。
2. **不**自动调用生成。
3. 若下游本地输入为空，可将「主文本参考」摘要预填到输入框（可选，且用户一改就不再覆盖）——实现计划定细节；产品底线是芯片必须出现。



### 6.2 上游内容变化

- 上游重新生成后，下游芯片 preview/payload **下次解析即更新**（因不落拷贝）。  
- 不自动重跑下游。



### 6.3 生成边界

- 只影响**当前被点生成的节点**。  
- `prompt` 整篇 → 下游：一次注入完整 `content`。

## 7. 非功能


| 项    | 要求                                           |
| ---- | -------------------------------------------- |
| 积分   | 下游每次成功发起生成扣点；引用本身不扣点                         |
| 大文件  | 沿用现有上传限制；失败有明确错误                             |
| 引用失效 | 上游删除/无 url：芯片「失效」+ 可移除                       |
| 持久化  | edges + 节点 `localRefs` / `refOrder`；正文仍在上游节点 |
| 超时   | 各类型沿用现有（文本/prompt 90s 等）                     |
| 鉴权   | 生成仍需登录                                       |

## 8. 范围拆分建议（审阅后据此写开发计划）

> 本节是**计划建议**，待本规格确认后再出详细 `plans/*.md` 任务拆解。



### P0 — 契约与解析层

- 统一 `resolveNodeRefs(targetId)`，取代「只取第一张图 / 本地非空忽略上游」。  
- 支持多文本、多图；`prompt` 优先 `content`。  
- edge 删除与芯片 × 同步。  
- 生成请求携带结构化 `refs[]`（`refKey` 如 `T1`）。

### P1 — Dock RefChip UI（全 Dock 面板）

- **单行**参考条、类型图标 + `T/I/V/A` 序号、拖拽排序、`refOrder`。  
- 上传/资产 → `localRefs` 与芯片同源。  

### P1.5 — 文本智能合并

- 服务端（或 agent）`mergeRefsToPrompt`：多文本源时 Chat 归纳，再进原生成链路。  
- 单源跳过合并。

### P1.1 — `@` 序号引用（可选紧随）

- 输入 `@T1` 高亮与展开。

### P2 — 图 → 文本视觉方案

- 文本生成 API 支持多图输入 / 视觉模型。  
- 默认电商方案类 system 模板（可配置）；积分与普通文本同价。



### P3 — 后续能力（仅登记）

- 视→图抽帧、视→文、音→文/音/视频、可连性强制矩阵、环检测、`prompt` 段落点选灌入。

## 10. 已拍板决策（含依据）

| 项 | 结论 | 决策依据 |
|---|---|---|
| 序号 | `T1/I1/V1/A1` | 类型可区分、避免跨类型抢「1」；后端用 `refKey` 解析，与 UI 单行无关 |
| 参考区布局 | **单行横滑** | 少占 Dock 高度；类型靠图标+前缀表达即可 |
| 图→文本积分 | 与普通文本同价 | 降低产品摩擦；视觉成本先由平台消化 |
| 音频对图片芯片 | 一期只展示 | 音频链路暂不消费参考图，避免假能力 |
| 文本多源 | Chat **归纳合并**，非朴素拼接 | 多段上游需语义整合后再生成，简单拼接效果差 |
| **上游 → `prompt`** | **B：长文不进模式生成上下文** | 见下方专条 |

### 上游 → `prompt` = B（依据）

**结论：** 上游文本芯片不进入 `prompt` 节点的 Call-1/Call-2；生成只使用本节点短字段 `prompt`。芯片可展示；仅短需求为空时允许摘要预填一次。

**依据：**

1. **`prompt` 职责单一**：短需求 → 模式扩写成长文；若再吃上游长文，等于「扩写的扩写」，易套娃、跑偏、变慢、费积分。  
2. **与下游灌入区分**：`prompt` 的 `content` 整篇注入 **image/text/video/audio** 是主价值；反向灌回 `prompt` 生成链路不是主路径。  
3. **用户仍可控**：需要借用上游文案时，预填到短框后可手改；或断开后在短框自行粘贴要点。  
4. **后续可再开**：若实证需要「基于旧提示词再变体」，再单列「变体模式」或显式开关，而不是默认吞长文。
